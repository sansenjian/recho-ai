# recho-ai 后端架构分析：Node.js 网关、Go 网关与微服务

## 执行摘要

recho-ai 当前后端包含两个并行网关：一个基于 Node.js + Express（`backend/gateway/`），一个基于 Go（`backend/go-gateway/`）。Node 网关仍是默认入口，负责聊天流、MCP 工具调用、技能加载、管理后台等复杂逻辑；Go 网关作为性能优化补充，正在逐步承接图片生成、历史记录、额度查询等冷启动敏感型路由。

这不是微服务架构，而是一种"模块化单体 + 双网关分流"的过渡形态。短期最务实的方向是继续收敛 Node 网关能力到 Go 网关，形成单一 Go 网关；真正引入微服务的时机取决于团队规模扩大、子域边界清晰、以及独立部署/扩缩容需求变得不可忽视。

## 当前架构快照

### 项目结构

```text
backend/
├── gateway/                 # Node.js + Express 网关（当前主入口）
│   ├── src/
│   │   ├── routes/          # /api/chat、/api/image/*、admin/*、tools 等
│   │   ├── services/        # chat-loop、image-api、credits、admin 等业务逻辑
│   │   ├── mcp/             # MCP 连接管理器与 OpenAI 工具 schema 适配
│   │   └── clients/         # Supabase、Tencent COS、LLM 客户端工厂
│   └── skills/              # 内置技能定义
└── go-gateway/              # Go 高性能网关（迁移目标）
    ├── cmd/server/main.go
    ├── internal/handler/    # HTTP 处理器
    ├── internal/service/    # 业务逻辑
    ├── internal/repository/ # 数据访问
    └── internal/middleware/ # 认证等中间件
```

### 前端流量走向

本地开发时，Vite 默认把 `/api` 代理到 Node 网关（`3000`）。若设置 `VITE_IMAGE_API_BASE_URL=http://127.0.0.1:3001`，图片相关调用会指向 Go 网关。生产环境中 Render 可同时部署两个服务：`recho-gateway`（Node）和 `recho-go-gateway`（Go），前端按路由分别访问。

## Node.js 网关 vs Go 网关：能力矩阵

| 能力 | Node.js 网关 | Go 网关 | 备注 |
|------|-------------|---------|------|
| 聊天流式 `/api/chat` | 完整实现，含 MCP 工具循环 | 仅基础流式转发 | Node 有 `chat-loop.ts` 和 `mcp/manager.ts` |
| MCP 工具调用 | 完整实现 | 未实现 | Go 侧缺少工具 schema 适配与调用循环 |
| 技能系统 `/api/skills` | 完整实现 | 未实现 | 技能加载与提示词注入在 Node |
| 图片生成 `/api/image/generate` | 完整实现 | 已实现 | Go 已作为迁移目标 |
| 图片历史 `/api/image/history` | 完整实现 | 已实现 | 含 public / mine 两种 scope |
| 额度系统 `/api/credits` | 完整实现 | 部分实现 | Go 侧有 credits / redeem |
| 管理后台 `/api/admin/*` | 完整实现 | 未实现 | 用户权限、图片审核、公告等 |
| 应用配置 `/api/config/*` | 完整实现 | 部分实现 | Go 侧可读取 app_settings |
| 图片存储代理 | 完整实现 | 待验证 | COS 行为与公开作品访问需对齐 |
| 冷启动 | 3-10s | 50-200ms | Render 无服务器场景差异显著 |
| 内存占用 | 100-200MB | 30-50MB | 图片来自 Go 网关 README |
| 测试覆盖 | 较完整 | 基础 | Node 侧测试数量明显更多 |

## 三种候选架构对比

### 方案 A：维持 Node.js 单网关

**优点**
- 一套技术栈，开发、部署、排障成本低。
- 聊天、MCP、技能、管理后台等复杂逻辑已经成熟，无需重写。

**缺点**
- Render 冷启动慢，内存占用高，无服务器账单压力较大。
- 图片生成这类对启动速度敏感的路径被拖累。

**适用场景**：团队小、聊天/MCP 迭代频繁、对成本不敏感。

### 方案 B：Go 网关为主，Node 网关为侧车（当前方向）

**优点**
- 图片、额度等高频低延迟路由获得 Go 的启动和内存优势。
- 聊天/MCP 等复杂逻辑保留在 Node，避免高风险重写。
- 迁移风险可控，可按路由逐步切换。

**缺点**
- 两套网关并存，鉴权、错误格式、数据库访问逻辑需要保持一致。
- 部署、监控、日志链路复杂度上升。
- 长期处于"半迁移"状态容易积累技术债。

**适用场景**：当前 recho-ai 所处阶段，性能痛点明确但复杂逻辑尚未稳定。

### 方案 C：按业务域拆分为微服务

**优点**
- 各服务独立部署、独立扩缩容，资源利用率更高。
- 多团队协作时天然减少代码冲突。
- 聊天、图片、管理后台可采用不同技术栈。

**缺点**
- 引入分布式事务、服务发现、链路追踪、事件总线等基础设施。
- recho-ai 当前业务域之间数据强耦合（用户、额度、图片记录共享度高），拆分成本高。
- 团队规模不足时，维护多个服务的收益无法覆盖开销。

**适用场景**：团队扩张到多个独立小组、各子域数据边界清晰、负载特征差异巨大。

## 微服务的适用条件

引入微服务前应同时满足以下多数条件：

1. **组织规模**：多个团队需要并行开发不同业务域，单一代码库已成为协作瓶颈。
2. **业务边界清晰**：聊天、图片、管理后台的数据和发布节奏可以自然分离，跨域调用少于域内调用。
3. **负载差异显著**：某些路径需要 GPU/高 IO，另一些只需要轻量 CPU，单体内部资源争抢严重。
4. **部署节奏不同**：聊天需要日更，图片需要周更，管理后台月更——耦合部署会互相阻塞。
5. **已有成熟 DevOps**：具备服务发现、可观测性、灰度发布、熔断降级等基础设施。
6. **性能瓶颈在架构层面**：单体优化（缓存、连接池、异步队列）已无法解决。

**当前 recho-ai 的匹配度**

| 条件 | 当前状态 |
|------|---------|
| 多团队协作 | 不满足，业务逻辑高度集中 |
| 业务边界清晰 | 部分满足，聊天/图片/管理职责分明但数据耦合强 |
| 负载差异 | 部分满足，图片生成与聊天流式特征不同 |
| 部署节奏差异 | 不明显 |
| DevOps 成熟度 | 未体现完整微服务基础设施 |
| 单体优化空间 | 仍有，Go 网关迁移本身就是优化 |

结论：recho-ai 当前**不满足引入微服务的典型条件**。

## 推荐路径

### 短期（当前 - 未来 1-2 个月）：继续双网关收敛

1. **把图片相关 API 全部切到 Go 网关**
   - `/api/image/history/*`
   - `/api/image/generate`
   - `/api/image/references`
   - 删除 Node 侧对应 handler，或标记为 deprecated。

2. **验证 Go 侧与 Node 侧的响应等价性**
   - 错误码、错误消息结构、鉴权失败行为必须一致。
   - 公开作品访问权限、COS 存储 URL 生成逻辑逐一对照。

3. **保留 Node 网关负责以下能力**
   - 聊天流 + MCP 工具循环
   - 技能加载与提示词注入
   - 管理后台 API
   - 工具端点 `/api/tools`

### 中期（未来 2-6 个月）：评估 Node 网关剩余能力迁移

- 如果 MCP 工具调用逻辑进入稳定期，可把 `backend/gateway/src/mcp/manager.ts` 的能力在 Go 侧重写。
- 如果管理后台需求增加，优先把只读/轻量管理接口迁到 Go；写操作和复杂审核逻辑保留 Node。
- 引入统一网关层（如 Traefik / Nginx / API Gateway），由它根据路径路由到 Node 或 Go，前端不再感知多个 base URL。

### 长期（6 个月后）：二选一

- **路径 A（推荐）**：Go 网关完全替代 Node 网关，下线 Node 服务，形成单一 Go 网关 + 模块化代码结构。
- **路径 B（备选）**：如果聊天/MCP 功能持续快速迭代且对 Go 重写不划算，保留"Go 高性能路由 + Node 复杂逻辑路由"的双服务结构，但把边界固化、文档化、监控补齐。

### 暂不建议的方向

- 直接按聊天、图片、管理后台拆成三个独立微服务：当前数据耦合度和团队规模会让拆分得不偿失。
- 为引入微服务而引入 Kubernetes、服务网格等重型基础设施：在业务未充分分裂前属于过度设计。

## 风险与注意事项

1. **数据一致性**：图片生成涉及额度扣减、记录写入、事件上报多个步骤，Node 与 Go 两侧实现必须原子语义一致。
2. **鉴权边界**：Supabase JWT 校验逻辑在 Node（`request-auth.ts`）和 Go（`middleware/auth.go`）中各有一份，权限规则变更需同步修改。
3. **错误契约**：前端对错误格式有预期，两侧返回结构不一致会导致用户提示失效。
4. **测试覆盖**：Go 侧测试数量少于 Node，迁移过程中需补充单元测试和接口契约测试。
5. **部署复杂度**：双服务部署会增加 Render/Vercel 配置量，需确保环境变量、健康检查、日志汇总一致。

## 结论

recho-ai 当前最合理的架构演进方向是：**用 Go 网关逐步替代 Node 网关，最终形成单一 Go 网关的模块化单体，而非直接拆微服务。** 微服务是团队协作和系统规模达到特定阈值后的选项，目前项目的核心矛盾是"Node 网关性能差"，这个矛盾通过 Go 网关迁移已经可以得到有效解决。
